Decisión de Tecnologías

Supabase (PostgreSQL)

Elegimos Supabase porque nuestro modelo de datos tiene relaciones complejas entre productos, categorías, proveedores y ventas, lo que requiere integridad referencial. Supabase expone una API REST automática sin necesidad de backend propio. 500 MB gratuitos y sin tarjeta de crédito.

GitHub Pages

Hosting gratuito, sin configuración de servidor y con disponibilidad 24/7. Al ser un frontend de JavaScript puro, no necesitamos ejecución de servidor. El deployment es automático al hacer push a la rama main.

HTML · CSS · JS Puro

Sin frameworks ni librerías externas. El código es 100% vanilla JS con fetch() para consumir la API de Supabase. Esto elimina dependencias, reduce el tiempo de carga y hace el proyecto más fácil de mantener.

Por qué no MongoDB Atlas

Nuestro modelo relacional de la Práctica 1 tiene FK, jerarquías y relaciones N:M bien definidas. Una base de datos NoSQL no puede expresar estas restricciones de integridad de forma nativa. PostgreSQL es la opción correcta para este caso.

Conclusión: Para este proyecto elegimos GitHub Pages porque el frontend es 100% estático y se comunica directamente con Supabase mediante fetch(). No necesitamos Render porque Supabase ya actúa como backend. Si en el futuro se requiriera lógica de servidor personalizada (autenticación propia, procesamiento de imágenes, webhooks), migraríamos el backend a Render.
